## Repository Description
- `helm-charts` contains Helm charts used to deploy Platform Mesh and supporting components.
- The main moving parts are charts under `charts/`, generated chart documentation, OCM constructor descriptors under `.ocm/`, local developer setup under `local-setup/`, and GitHub Actions workflows under `.github/workflows/`.
- Read the org-wide [AGENTS.md](https://github.com/platform-mesh/.github/blob/main/AGENTS.md) for general conventions.

## Core Principles
- Keep chart changes small and reviewable. Values, templates, chart versions, dependencies, and generated docs are tightly coupled.
- Prefer existing `task` workflows over ad-hoc Helm commands.
- Update generated chart docs and dependencies when chart inputs change.
- Keep this file focused on agent execution and repository-specific constraints.

## Project Structure
- `charts/`: Helm charts, values, templates, tests, chart locks, and chart READMEs.
- `docs-templates/`: templates used by `helm-docs` to generate chart documentation.
- `.ocm/`: OCM constructor descriptors for packaging charts and components.
- `local-setup/`: scripts, manifests, kustomize overlays, and e2e assets for local Platform Mesh development.
- `.github/workflows/`: chart validation, publishing, local setup, and component workflows.
- `Taskfile.yaml`: primary local automation entrypoint.

## Architecture
This is a deployment and packaging repo, not a service runtime.

### Chart model
- Each chart under `charts/` owns its `Chart.yaml`, `values.yaml`, templates, optional test values, and generated README.
- Several charts depend on other charts in this repository; chart version bumps and dependency updates need to stay aligned.
- `helm-docs` generates chart README content from chart metadata, values, and templates.

### Local setup and OCM model
- `local-setup/` is the supported path for bootstrapping and testing a local Platform Mesh instance.
- `.ocm/` and related `task ocm:*` targets build and transfer OCM component artifacts for local and release workflows.
- Workflow changes can affect chart publishing and full platform bootstrap behavior.

## Commands
- `task lint` — run chart-testing lint against charts.
- `task test` — run Helm unit tests for all charts.
- `task helmtest-update` — update Helm unittest snapshots when expected output changes.
- `task docs` — regenerate chart documentation with `helm-docs`.
- `task update-changed` — regenerate docs, update dependencies, and update snapshots for changed charts.
- `task update` — regenerate docs and update dependencies for all charts.
- `task validate` — run test, docs, dependency update, lint, and OCI packaging checks.
- `task vulnerability` — run kube-linter against charts.
- `task oci` — package charts into the local `oci/` output directory.

## Code Conventions
- Bump the chart version for any chart whose rendered behavior changes.
- Run `task docs` after changing chart metadata, values, README templates, or generated chart docs.
- Run `task update-changed` after changing chart dependencies or values that affect snapshots.
- Keep templates readable; prefer chart-local helper templates over duplicated complex logic.
- Keep secrets and credentials out of values files, workflows, and local setup scripts.

## Generated Artifacts
- Chart READMEs are generated by `task docs`; do not hand-edit generated sections when a source value/template change is required.
- `Chart.lock` files should be updated through Helm dependency workflows.
- Helm unittest snapshots should be updated through `task helmtest-update` or `task update-changed`.

## Do Not
- Skip chart version bumps for behavior-changing chart edits.
- Hand-edit generated chart documentation as a substitute for changing chart metadata or values.
- Commit local secrets from `.secret/`, generated binaries from `bin/`, or local OCI output.
- Introduce new chart publishing or local setup paths when existing `task` targets cover the workflow.

## Hard Boundaries
- Ask before changing chart publishing, OCM packaging, or local setup bootstrap semantics.
- Be especially careful with changes under `.github/workflows/`, `.ocm/`, and `local-setup/`.
